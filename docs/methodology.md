# Methodology Notes

## Embodied Debrief

Embodied Debrief is the working method for returning to a completed production through its residues: notes, chats, schedules, self-tracking data, technical snapshots, material remnants, and installation design.

The method is not only an archive. It is a structured re-reading of a production process through traces that normally remain secondary to the final artwork.

## Corpus Logic

The corpus should preserve enough material to make the analysis reproducible:

- what was collected;
- how time, money, attention, and tasks were allocated;
- where work was delegated to people, tools, AI systems, and infrastructure;
- where overload became visible in records, decisions, delays, or bodily traces.

The repository should therefore contain both selected traces and the reasoning that explains why they matter.

## Analytical Modes

### Collect

Material that shows accumulation: reference piles, notes, exported contexts, screenshots, unresolved ideas, logs, lists, and gathered media.

### Allocate

Material that shows distribution: schedules, budgets, task boards, production plans, delegation calendars, spatial layouts, and decisions about resources.

### Delegate

Material that shows externalised agency: collaborator communication, AI exchanges, software automation, TouchDesigner logic, scripts, hardware control, and outsourced decisions.

### Overload

Material that shows limits: plan saturation, repeated revisions, sleep/body data, health notes, delays, conflicts, emergency fixes, and moments where the system exceeded available capacity.

## Evidence Levels

Use three levels when writing notes or derived summaries:

| Level | Description |
| --- | --- |
| `direct_trace` | The claim is grounded in a specific file or excerpt. |
| `derived_pattern` | The claim comes from a summary, count, timeline, or comparison. |
| `reflective_interpretation` | The claim is a later interpretation by the author. |

Good research notes should make this distinction explicit.

## Reproducibility

For each processed output, preserve:

- source file id from `data/manifest.csv`;
- script or manual process used;
- date of processing;
- redaction notes;
- resulting file path.

This makes it possible to return from a thesis argument to the trace that supports it, without exposing private raw material unnecessarily.
